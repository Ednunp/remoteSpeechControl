Remote Speech Control

Two related features for NVDA Remote sessions:

  1. Speech muting on the controlled machine. Silences NVDA's local speech on a machine that is being controlled by NVDA Remote, while a remote keyboard is driving NVDA. The moment a real keypress is detected on the controlled machine, speech resumes; it mutes again on the next remote keystroke. Muting is gated by a shared password, by explicit per-session consent from the controlled side's user, and (while the consent prompt is pending) by a freeze on all remote input.
  2. Local synth settings ring. Optionally keeps NVDA's synth settings ring (rate, pitch, volume etc.) on the controller's local NVDA while you are in remote control mode, instead of forwarding those keystrokes to the remote.

Why this exists

The problem this add-on solves is doubled audio when you control one machine from another with NVDA Remote and you also have some way of streaming the controlled machine's sound back to where you are sitting. That second channel can be a low-latency audio-over-IP tool, Windows' own Sound Forwarding, a virtual audio cable — anything that mirrors the controlled machine's playback to your local speakers.

In that arrangement NVDA Remote already forwards speech to your controlling NVDA, so your local synth speaks every announcement. At the same time, the controlled machine's NVDA is also speaking through its own audio device, and the audio-streaming setup is faithfully relaying that to you a fraction of a second later. The two voices overlap, slightly out of sync, and the controlled machine becomes unusable to drive remotely.

You could just change the controlled machine's NVDA output to a silent or virtual audio device, but it is far too easy to forget to switch it back. Arriving at the controlled machine later and finding no speech at all is worse than the doubled audio you were trying to fix in the first place.

This add-on silences the controlled machine's local NVDA only while you are remotely driving it. The moment you press a key on the controlled machine itself, speech resumes immediately. There is nothing to remember to switch back, because there is no manual switch — only your own keystroke on the local keyboard turns speech off and on again.

The companion feature, keeping NVDA's synth settings ring on your controlling machine, addresses a related annoyance. When NVDA Remote is forwarding speech, the synth you actually hear is your local one. Adjusting rate, pitch, volume or voice ought to change your local NVDA, not the remote NVDA you can no longer hear. Without this add-on, those gestures are forwarded to the remote and adjust nothing audible.

Setting up

Install the add-on on both machines. Open the NVDA menu, choose Preferences, then Settings, then the Remote Speech Control category. Type the same password into the Password field on both ends. The password is never transmitted over the network; only a signed proof of it is, so an attacker who captures network traffic cannot recover it.

If you regularly control the same two machines from each other, you can pre-consent on each side using the two checkboxes described below. By default both are unticked, which means every mute requires an explicit yes from each side.

Settings

Password

Shared secret between the two machines. It must match exactly on both ends, including case. Without a password on either side, the mute feature does nothing.

Auto-request mute when I connect as controller (Alt+A)

Controller-side setting. When ticked, your machine automatically sends a mute request to the controlled machine the moment a remote session connects, instead of popping a yes/no prompt on your side first. Default unticked.

Allow speech to be automatically muted by controlling machine (Alt+M)

Controlled-side setting. When ticked, your machine accepts authenticated mute requests without prompting you, and without freezing remote input. When unticked, you get a yes/no dialog asking whether to allow muting, all remote input is paused until you answer, and nothing is muted until you accept. Default unticked.

Synth settings ring adjusts this machine, not the remote (Alt+S)

Controller-side setting. When ticked, pressing NVDA's synth settings ring gestures (move between settings, increase, decrease) while in remote control mode adjusts the local NVDA's voice, rate, pitch, volume and so on, rather than being forwarded to the remote. Default unticked.

Check for updates daily (Alt+U)

When ticked, the add-on quietly checks GitHub once a day for a newer release and prompts you to download and install it if one is available. Default unticked.

Check for updates now (Alt+C)

A button rather than a setting. Click it to make the add-on check for a newer release immediately, regardless of whether the daily check is enabled. If you previously declined an update, this button will offer it again.

Verbose logging (Alt+V)

Writes detailed activity into NVDA's main log. Useful for diagnosing problems. Default ticked.

Using the mute feature

Connect with NVDA Remote as usual. Within a second or two of the session coming up:

  • On the controlling machine, you will be asked "Mute speech on the remote machine while you control it from here?", unless you ticked Auto-request mute, in which case the request goes straight away.
  • On the controlled machine, you will be asked "The controller has requested to mute speech on this machine. Remote input is paused until you answer. Allow muting for this session?", unless you ticked Allow speech to be automatically muted, in which case the request is accepted straight away without freezing input.

Once both sides accept, NVDA's local speech on the controlled machine is dropped for as long as input is coming from the controller's keyboard. The moment any key is pressed on the controlled machine itself, speech resumes immediately — until the next remote keystroke, which mutes it again. This is what is meant by "ping-pong".

Disconnecting the NVDA Remote session always unmutes the controlled machine automatically.

The remote-input freeze during consent

While the controlled machine is waiting for the local user to answer the consent prompt, all remote keyboard and braille input from the controller is paused on the controlled machine. This is the security property that distinguishes consent from a yes/no the controller can simply press themselves. Only the local user's physical keyboard (or other locally-attached input) can resolve the prompt.

The controller's NVDA will announce "Waiting for remote user to allow mute. Remote input is paused." when the freeze begins. After the local user answers, input flows again — and you will hear the result announced ("Remote machine speech muted" or "Remote machine declined the mute request").

If you want to skip the freeze entirely, tick "Allow speech to be automatically muted by controlling machine" on the controlled machine. With that ticked, authenticated mute requests apply immediately and no consent prompt or input freeze ever happens.

Force-unmute

If you arrive at the controlled machine and want to fully release the mute, so that the controller would have to make a new request before muting again, press NVDA+shift+u. This unmutes immediately and forgets the consent you gave for this session.

The shortcut is rebindable. Open the NVDA menu, choose Preferences, then Input gestures, then look under the Remote Speech Control category.

Synth settings ring on the local machine

NVDA's synth settings ring (default gestures around NVDA+control or NVDA+control+shift with the arrow keys, depending on your configuration) lets you cycle through synthesiser settings and adjust them. Without this add-on, those keystrokes are forwarded to the remote machine while you are in remote control mode — useless because you are listening to your local speech, not the remote's.

Tick "Synth settings ring adjusts this machine, not the remote" on the controller side and those specific gestures stay local. All other keystrokes still forward to the remote normally. NVDA Remote already had a mechanism for keeping specific scripts local; we just register the four ring scripts (plus the two large-step variants) with it.

Automatic updates

If you tick "Check for updates daily", the add-on contacts api.github.com once every 24 hours, asks for the most recent published release, and compares it to the version you have installed. If the release is newer, a yes/no dialog appears: download and install it now? If you click yes, the add-on downloads the new .nvda-addon file straight from the GitHub release, installs it, and prompts you to restart NVDA. If you click no, the same version is not offered again automatically; the next time a newer version appears, you will be asked again.

The Check for updates now button on the settings panel triggers the same check immediately and ignores any previous decline, so you can use it to revisit a version you said no to earlier.

This is independent of NVDA's own Add-on Store, which has its own update mechanism for add-ons installed via the Store. If you installed via the Store, both update paths will be available; whichever notices a new version first wins.

Security

The add-on proves the two ends are paired using a shared password. The password is fed through PBKDF2-HMAC-SHA256 to derive a key, which is used to sign each mute request with an HMAC. The signature includes a random nonce and a timestamp, both of which are checked on receipt to prevent replay attacks. After three failed authentication attempts on the same transport, further requests are silently dropped for sixty seconds.

If a controller can connect to your NVDA Remote channel but does not have the password set, they cannot mute your speech. If they have the password but you have not ticked Allow speech to be automatically muted, you will still be prompted before any mute takes effect — and crucially, the controller cannot answer that prompt themselves, because remote input is frozen on your machine until you do.

Troubleshooting

The add-on writes its progress and errors into NVDA's main log. Open the NVDA menu, choose Tools, then View log. Search for lines beginning rsc:.

On a working session you should see, on each machine, lines like "starting up", "wrapped synth speak", "WH_KEYBOARD_LL hook installed", and "_remoteClient integration installed". When a session connects you should see "transport connected" and "announcing capability". When the two sides have exchanged capability messages you should see "peer capability received".

If muting never happens, the most common cause is the password being different on the two machines. Set it again on both ends and try.
